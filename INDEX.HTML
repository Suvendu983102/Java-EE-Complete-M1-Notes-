Introduction to Java EE

Java EE (Java Platform, Enterprise Edition), now known as Jakarta EE, is a set of specifications extending Java SE with enterprise features such as distributed computing and web services. It provides a robust platform for developing and running large-scale, multi-tiered, scalable, reliable, and secure network applications.

Platform Independent

Write once, run anywhere - Java's core principle applies to Java EE applications too.

Component-Based

Modular design with reusable components promotes maintainability.

Container-Managed

Containers handle component lifecycle, security, transactions, and more.

Declarative

Configuration through annotations and XML reduces boilerplate code.

Key Takeaway

Java EE is not a product but a standard specification. Different vendors like Oracle, Red Hat, and Eclipse provide implementations of these specifications (like GlassFish, WildFly, TomEE).

Java EE Architecture

Multi-Tier Architecture

Java EE applications typically follow a multi-tier architecture that separates concerns into distinct layers:

1. Client Tier (Presentation Layer)

Web browsers, mobile devices, or desktop applications that handle user interface and interaction. Can be thin clients (HTML/JavaScript) or thick clients (standalone applications).

2. Web Tier (Presentation Logic)

Handles HTTP requests and generates dynamic content. Technologies include:

  • Servlets: Handle HTTP requests and responses
  • JSP (JavaServer Pages): Dynamic web page generation
  • JSF (JavaServer Faces): Component-based UI framework

3. Business Tier (Business Logic Layer)

Contains the core business logic of the application:

  • EJB (Enterprise JavaBeans): Business logic components
  • Session Beans: Execute business logic
  • Message-Driven Beans: Asynchronous message processing
  • Web Services: RESTful and SOAP endpoints

4. Data Tier (Enterprise Information System)

Handles data persistence and storage:

  • Database systems: MySQL, PostgreSQL, Oracle
  • JPA (Java Persistence API): Object-relational mapping
  • JDBC: Database connectivity
  • File systems and messaging systems

// Example of a simple Java EE architecture flow
Client (Browser) → Web Tier (Servlet/JSP) →
Business Tier (EJB/Service Classes) → Data Tier (JDBC/JPA) → Database

Java SE vs Java EE

Understanding the difference between Java SE (Standard Edition) and Java EE (Enterprise Edition) is crucial for choosing the right platform for your application.

Java SE (Standard Edition)

The foundation platform for Java development. Provides core Java functionality suitable for desktop and standalone applications.

Key Features:

  • Core language features: OOP, collections, exceptions, generics
  • Basic libraries: java.lang, java.util, java.io, java.nio
  • GUI development: Swing, AWT (though less common today)
  • Networking: java.net package for network programming
  • Multithreading: java.util.concurrent for concurrent programming
  • JDBC: Basic database connectivity

Java EE (Enterprise Edition)

Built on top of Java SE. Designed for large-scale, distributed, enterprise applications with comprehensive infrastructure for web and enterprise applications.

Key Features (Includes all Java SE features plus):

  • Web application development: Servlets, JSP, JSF, JSTL
  • Enterprise JavaBeans (EJB): Business logic components
  • Web services: JAX-RS (RESTful), JAX-WS (SOAP)
  • Messaging services: JMS (Java Message Service)
  • Transaction management: JTA (Java Transaction API)
  • Persistence: JPA (Java Persistence API)
  • Security: JAAS (Java Authentication and Authorization Service)
  • Dependency Injection: CDI (Contexts and Dependency Injection)

Comparison Summary

Java SE: Core Java platform for general-purpose applications (desktop, command-line).
Java EE: Enterprise extension for building large-scale, distributed, web-based applications.

Think of Java SE as the foundation, and Java EE as a specialized toolkit built on that foundation for enterprise needs.

JDBC in Web Applications

What is JDBC?

JDBC (Java Database Connectivity) is a Java API that enables Java applications to interact with relational databases. It provides a standard interface for connecting to databases, executing SQL queries, and processing results.

JDBC Architecture

JDBC uses a driver manager and database-specific drivers to communicate with databases:

Java Application → JDBC API → JDBC Driver Manager →
JDBC Driver → Database

JDBC Driver Types

  1. Type 1: JDBC-ODBC Bridge Driver - Translates JDBC calls to ODBC calls (deprecated)
  2. Type 2: Native-API Driver - Converts JDBC calls to database-specific native calls
  3. Type 3: Network Protocol Driver - Pure Java driver that sends JDBC calls to middleware server
  4. Type 4: Thin Driver (Most Common) - Pure Java implementation that converts JDBC calls directly to database protocol

Basic JDBC Workflow

// 1. Load the JDBC driver
Class.forName("com.mysql.cj.jdbc.Driver");

// 2. Establish a connection
Connection conn = DriverManager.getConnection(
    "jdbc:mysql://localhost:3306/mydb", "username", "password");

// 3. Create a statement
Statement stmt = conn.createStatement();

// 4. Execute a query
ResultSet rs = stmt.executeQuery("SELECT * FROM users");

// 5. Process the results
while(rs.next()) {
    String name = rs.getString("username");
    System.out.println(name);
}

// 6. Close resources
rs.close();
stmt.close();
conn.close();

Important Note

Always use PreparedStatement instead of Statement for queries with parameters to prevent SQL injection attacks. Also, always close database resources (Connection, Statement, ResultSet) in a finally block or use try-with-resources to avoid memory leaks.

JavaServer Pages (JSP)

What is JSP?

JSP (JavaServer Pages) is a server-side technology used to create dynamic web pages by embedding Java code within HTML. JSP pages are compiled into servlets by the container, combining the power of Java with the simplicity of HTML.

JSP Lifecycle

  1. Translation Phase: JSP is converted to servlet source code (.java file)
  2. Compilation Phase: Servlet source is compiled to bytecode (.class file)
  3. Initialization Phase: jspInit() method is called once when the servlet is instantiated
  4. Request Processing: _jspService() method handles each client request
  5. Destruction Phase: jspDestroy() method is called when the servlet is unloaded

JSP Syntax Elements

<!-- 1. Directive Tags -->
<%@ page language="java" contentType="text/html" pageEncoding="UTF-8"%>
<%@ include file="header.jsp" %>

<!-- 2. Declaration Tags -->
<%!
    private int count = 0;
    public void increment() { count++; }
%>

<!-- 3. Scriptlet Tags -->
<%
    String name = request.getParameter("username");
    if(name != null) {
        out.println("Hello, " + name);
    }
%>

<!-- 4. Expression Tags -->
Current time: <%= new java.util.Date() %>

<!-- 5. Action Tags -->
<jsp:include page="footer.jsp" />
<jsp:forward page="result.jsp" />

JSP Implicit Objects

JSP provides several implicit objects (predefined variables) that can be used without declaration:

  • request: HttpServletRequest object containing client request information
  • response: HttpServletResponse object for sending response to client
  • out: JspWriter object for writing content to output stream
  • session: HttpSession object for maintaining user session data
  • application: ServletContext object shared across entire application
  • pageContext: PageContext object providing access to all implicit objects
  • config: ServletConfig object for servlet configuration
  • page: Reference to the current servlet instance (like 'this')
  • exception: Throwable object available in error pages

Best Practice

Minimize Java code in JSP pages by using JSTL (JSP Standard Tag Library) and EL (Expression Language). This separates presentation logic from business logic and makes JSP pages cleaner and more maintainable.

Servlets

What is a Servlet?

A servlet is a Java class that runs on a web server and handles HTTP requests and responses. Servlets are the foundation of Java web applications, providing a robust, scalable way to build server-side components.

Servlet Lifecycle

  1. Loading: Container loads servlet class (typically on first request)
  2. Initialization: init() method is called once with ServletConfig parameter
  3. Request Handling: service() method dispatches requests to appropriate doGet(), doPost(), etc.
  4. Destruction: destroy() method is called before servlet is unloaded from memory

Servlet Example

import java.io.*;
import javax.servlet.*;
import javax.servlet.http.*;

public class HelloServlet extends HttpServlet {

    // Handle GET requests
    protected void doGet(HttpServletRequest request,
        HttpServletResponse response) throws ServletException, IOException {
        response.setContentType("text/html");
        PrintWriter out = response.getWriter();
        out.println("<html><body>");
        out.println("<h1>Hello from Servlet!</h1>");
        out.println("</body></html>");
    }

    // Handle POST requests
    protected void doPost(HttpServletRequest request,
        HttpServletResponse response) throws ServletException, IOException {
        String name = request.getParameter("username");
        // Process form submission...
    }
}

Servlet Configuration

Servlets can be configured in two ways:

// 1. Using web.xml (Deployment Descriptor)
<servlet>
    <servlet-name>HelloServlet</servlet-name>
    <servlet-class>com.example.HelloServlet</servlet-class>
</servlet>
<servlet-mapping>
    <servlet-name>HelloServlet</servlet-name>
    <url-pattern>/hello</url-pattern>
</servlet-mapping>

// 2. Using Annotations (Java EE 5+)
@WebServlet(
    name = "HelloServlet",
    urlPatterns = {"/hello", "/hello/*"},
    initParams = {
        @WebInitParam(name = "message", value = "Hello World")
    }
)
public class HelloServlet extends HttpServlet { ... }

Servlet Best Practices

  • Avoid using instance variables in servlets (they're not thread-safe)
  • Always close resources (database connections, file streams) in finally blocks
  • Use RequestDispatcher for forwarding to JSP pages
  • Implement proper error handling with custom error pages
  • Use filters for cross-cutting concerns (logging, authentication, encoding)

Technology Integration

How JDBC, JSP, and Servlets Work Together

In a typical Java EE web application, these technologies work together in a coordinated manner:

// 1. User submits form via JSP
<form action="RegisterServlet" method="post">
    <input type="text" name="username">
    <input type="submit" value="Register">
</form>

// 2. Servlet processes the request
@WebServlet("/RegisterServlet")
public class RegisterServlet extends HttpServlet {
    protected void doPost(...) {
        String username = request.getParameter("username");

        // 3. Use JDBC to save to database
        Connection conn = ...;
        PreparedStatement pstmt = conn.prepareStatement(
            "INSERT INTO users(username) VALUES(?)");
        pstmt.setString(1, username);
        pstmt.executeUpdate();

        // 4. Forward to result JSP
        request.setAttribute("message", "Registration successful!");
        RequestDispatcher rd = request.getRequestDispatcher("result.jsp");
        rd.forward(request, response);
    }
}

MVC Pattern Implementation

Java EE applications often follow the Model-View-Controller (MVC) pattern:

Model (Business Logic + Data)

  • JavaBeans/POJOs represent data
  • DAO (Data Access Object) classes handle JDBC operations
  • Service classes contain business logic

View (Presentation)

  • JSP pages display data
  • Use JSTL and EL for dynamic content
  • Minimal Java code in JSPs

Controller (Request Handling)

  • Servlets handle HTTP requests
  • Coordinate between Model and View
  • Manage application flow and navigation

Modern Java EE Development

Evolution to Jakarta EE

Java EE has transitioned to Jakarta EE under the Eclipse Foundation. Key changes include:

  • New namespace: jakarta.* instead of javax.*
  • Active community-driven development
  • Modern cloud-native features
  • Regular release cycles

Popular Frameworks and Tools

Spring Framework

Comprehensive enterprise framework with dependency injection, AOP, and extensive modules.

Spring Boot

Rapid development with embedded servers, auto-configuration, and production-ready features.

Hibernate/JPA

Object-relational mapping for simplified database operations.

Microservices

Architectural style for breaking applications into smaller, independently deployable services.

Learning Path Recommendations

  1. Start with core Java: Solid understanding of Java SE is essential
  2. Learn Servlets and JSP: Foundation of Java web development
  3. Master JDBC: Database connectivity is crucial for most applications
  4. Explore frameworks: Spring Boot is highly recommended for modern development
  5. Build projects: Apply knowledge by building complete applications
  6. Learn complementary technologies: HTML, CSS, JavaScript, SQL

Conclusion

Understanding Java EE architecture, JDBC, JSP, and Servlets provides a solid foundation for developing enterprise Java applications. While newer frameworks like Spring Boot have gained popularity, the core concepts remain relevant. Practice building applications that integrate these components to reinforce your learning and prepare for real-world development challenges.
